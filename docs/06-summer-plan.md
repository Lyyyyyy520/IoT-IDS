## 二、板块问题清单（当前真实状态）

### 板块 A：态势大屏 Dashboard

| 问题 | 严重度 | 描述 |
|------|--------|------|
| A1. 数据源半真半假 | 中 | KPI 卡片读 SQLite、但 traffic_history 和 attack_distribution 仍为硬编码 |
| A2. 无自动刷新 | 低 | 页面需手动刷新才更新数据 |
| A3. 风险评分算法简陋 | 低 | risk_score 只是简单减法，无加权 |

### 板块 B：告警中心

| 问题 | 严重度 | 描述 |
|------|--------|------|
| B1. block/trace/false-positive 是空壳 | **高** | 三个按钮只返回成功消息，不写库、不执行任何操作 |
| B2. 无实时弹窗 | 中 | 新告警不会主动通知，需手动刷新 |
| B3. 告警合并未实现 | 中 | 同源IP/同类型的告警不会聚合成一条 |
| B4. 批量操作缺失 | 低 | 无法一次标记多条为误报 |

### 板块 C：分析视图

| 问题 | 严重度 | 描述 |
|------|--------|------|
| C1. 三个 API 全部硬编码 | **高** | topology/heatmap/mitre 返回固定 Mock 数据，与真实告警无关 |
| C2. 拓扑图无交互 | 中 | 点击设备节点无反应 |
| C3. 热力图数据随机 | 中 | 每次刷新数据不同 |

### 板块 D：流量分析

| 问题 | 严重度 | 描述 |
|------|--------|------|
| D1. 仿真模式下攻击流量比例固定 | 低 | 25% 硬编码，不可配 |
| D2. 协议分布饼图数据为假 | 中 | 五个协议百分比写死 |

### 板块 E：三个缺失页面

| 问题 | 严重度 | 描述 |
|------|--------|------|
| E1. 策略管理页不存在 | **高** | 目录为空，IP 黑白名单功能完全缺失 |
| E2. 日志中心页不存在 | **高** | 目录为空，审计/告警/流量日志无法统一查询 |
| E3. 资产监控页不存在 | 中 | 目录为空，IoT 设备状态无法直观查看 |
| E4. 三个页面都无后端 API | **高** | 数据库有表，但无 CRUD 接口 |

### 板块 F：检测引擎

| 问题 | 严重度 | 描述 |
|------|--------|------|
| F1. 模型用合成数据训练 | **高** | 10,000 条 random 数据，非真实 IoT 流量，论文/答辩无法采信 |
| F2. 未做剪枝量化 | **高** | 原始模型 420KB，未压缩；树莓派上未做 INT8 优化 |
| F3. 无对比实验 | 中 | 论文必需的 SVM/RF/CNN/LSTM 对照数据完全缺失 |
| F4. 特征提取无真实 PCAP 验证 | 中 | extract_from_pcap 实际上是 mock_extract |

### 板块 G：树莓派探针

| 问题 | 严重度 | 描述 |
|------|--------|------|
| G1. 探针客户端是仿真 | **高** | probe_client.py 随机生成假告警，非 Suricata 真实输出 |
| G2. 未安装 Suricata | **高** | 树莓派上未实际部署任何 IDS 引擎 |
| G3. 无稳定性测试 | 中 | 未做过长时间运行测试 |

## 三、重新分工（按模块难度分配）

### 李云锦 — 后端核心 + 检测引擎集成（难度：高）

**负责板块**：B（告警后端）、F（检测引擎联调）、E4（策略/日志/资产 API）

| 周次 | 任务 |
|------|------|
| 第 1 周 | **B1 告警真实处置**：block→IP 写入 policies 黑名单表 + 返回成功；trace→关联查询同 IP 历史告警列表 + 返回结构化溯源信息；false-positive→更新 alert.status='false_positive' 并记审计日志。**E4 策略管理 API**：`GET/POST/DELETE /api/policies` 黑白名单增删查 |
| 第 2 周 | **E4 日志中心 API**：`GET /api/logs/audit`（审计日志分页+筛选）、`GET /api/logs/alerts`（告警日志分页+筛选）、`GET /api/logs/traffic`（流量日志分页+筛选）。**E4 资产 API**：`GET /api/assets`（列表）、`POST /api/assets/refresh`（更新在线状态） |
| 第 3 周 | **B3 告警合并**：定时任务或写入时检查同 src_ip + 同 attack_type + 5 分钟窗口→merged_count+1→不插新行。**F 规则引擎联调**：配合谢庚泉新增的规则，验证能命中并生成告警 |
| 第 4 周 | 全链路联调 + Bug 修复 + 代码 Review |

### 谢庚泉 — 数据集 + 规则引擎（难度：高）

**负责板块**：F（数据/规则）、D（流量数据真实化）

| 周次 | 任务 |
|------|------|
| 第 1 周 | **F1 真实数据集**：下载 N-BaIoT（~3.2GB）和 BoT-IoT 5% 子集（~1.1GB）。编写 Python 脚本按设备类型过滤（保留摄像头/门禁/传感器/家居/路由器 5 类），输出 train.csv + test.csv，含 21 维特征列 + label 列 |
| 第 2 周 | **F1 特征筛选验证**：在真实数据上重跑 46→21 维三层筛选，输出每层剔除的特征名称和原因。**规则扩充**：编写 10 条新规则——MQTT 异常订阅、CoAP 反射攻击、UPnP 漏洞利用、IoT 默认口令扫描、DNS 隧道检测等 |
| 第 3 周 | **D2 流量数据真实化**：让协议分布饼图从 traffic_logs 表读真实比例。**规则命中率验证**：用测试 PCAP 跑全部规则，统计每条命中次数，输出报告 |
| 第 4 周 | 实验数据汇总 + 规则文档 |

### 李津涛 — 模型训练与优化（难度：高）

**负责板块**：F（模型训练/压缩/对比实验）

| 周次 | 任务 |
|------|------|
| 第 1 周 | **F1 真实数据训练**：拿谢庚泉产出的训练集跑 train.py，调参使验证集准确率 ≥95% |
| 第 2 周 | **F2 模型压缩**：① 结构化剪枝——用 PyTorch prune 移除 50% 冗余权重→微调 5 epoch→记录精度/体积变化；② INT8 量化——用 PyTorch quantization 做量化感知训练→导出 ONNX→对比 FP32 vs INT8 的精度/体积/推理速度 |
| 第 3 周 | **F3 对比实验**：训练 SVM、随机森林、普通 CNN、普通 LSTM 四个对照模型，输出六列对比表（模型 | 准确率 | 召回率 | F1 | 参数量 | 推理时间） |
| 第 4 周 | 实验图表整理 + 最优 ONNX 模型交付树莓派 |

### 杨明敏 — 前端页面（难度：低—中）

**负责板块**：E（三个缺失页面）、A2（自动刷新）

| 周次 | 任务 |
|------|------|
| 第 1 周 | **E1 策略管理页**：两个表格——黑名单（IP/备注/添加时间/启用开关/删除按钮）、白名单（同上）。一个新建表单弹窗（IP 输入+备注+类型选择） |
| 第 2 周 | **E2 日志中心页**：Ant Design Tabs 三个 Tab——告警日志表（从 `/api/alerts` 读）、流量日志表（从 `/api/traffic/logs` 读）、审计日志表（从 `/api/logs/audit` 读）。每个 Tab 上方放时间范围和类型筛选下拉框。导出 Excel 按钮对接 `/api/export/excel` |
| 第 3 周 | **E3 资产监控页**：设备卡片列表——每张卡片显示设备名/IP/类型图标/状态点（绿在线/灰离线）/最后心跳时间。从 `/api/assets` 读数据。**A2 Dashboard 轮询**：`setInterval` 每 10 秒调一次 `/api/dashboard/stats` 自动刷新 |
| 第 4 周 | UI 统一走查——所有页面字号/间距/颜色一致。Bug 修复 |

### 韦思杨 — 树莓派部署 + 测试（难度：低—中）

**负责板块**：G（树莓派探针）、D1（抓包配置）

| 周次 | 任务 |
|------|------|
| 第 1 周 | **G 树莓派环境**：烧录 Raspberry Pi OS→安装 Python/ONNX Runtime→拷贝 best_model.onnx + edge_detect.py→运行 demo 模式确认推理成功。截图每步，为部署文档积累素材 |
| 第 2 周 | **G2 Suricata 安装**：`sudo apt install suricata`→配置监听网卡→确认 eve.json 正常输出告警。**G 适配脚本**：写 suricata_adapter.py——读 eve.json tail→解析 alert 事件→调 `/api/probe/push` 推送到管理端→验证管理端收到 |
| 第 3 周 | **G3 24h 测试**：树莓派通电跑 24h→每小时记录一次 CPU/内存/温度→第 2 天汇总测试报告（丢包率/误报数/资源占用）。**D1 抓包模式**：在 Windows 上装 Npcap→测试 Scapy sniff 能否抓到真实流量，记录结果 |
| 第 4 周 | 编写部署文档（树莓派从零 → 运行探针的完整图文步骤）。演示环境搭建 |

### 胡宇翔 — 可视化数据 + 文档（难度：低）

**负责板块**：C（分析视图数据真实化）、文档更新

| 周次 | 任务 |
|------|------|
| 第 1 周 | **C1 拓扑图改造**：前端从 `/api/analysis/topology` 拿数据保持不变，但让后端 API 根据 assets 表动态返回设备节点（去掉硬编码）。**拓扑图节点交互**：点击节点弹出小卡片显示 IP/MAC/类型/近 24h 告警数 |
| 第 2 周 | **C2 热力图改造**：前端从 `/api/analysis/heatmap` 拿数据，改后端 API 为按真实 alerts 表的 created_at 时间分布统计，返回 7 天×12 时段矩阵。**C3 MITRE 改造**：根据 alerts 表的 attack_type 判断哪些阶段 active（有对应类型告警→active=true） |
| 第 3 周 | **演示数据注入**：写脚本向 DB 注入 100+ 条分布合理的模拟告警（覆盖所有 attack_type × risk_level 组合），确保大屏/分析/告警页每个图表都有数据可看 |
| 第 4 周 | 项目 README 重写 + 所有 docs/ 更新至当前版本 + 整理最终成果文件结构 |
